# oh-my-tai

有时候你会盯着终端，脑子里知道自己以前写过一个小脚本，知道它大概能解决眼前这个问题，但就是想不起来名字、参数、路径，甚至不想为了这一下专门切出去翻历史记录。

这个小项目就是在那种时刻冒出来的。

起因很直接。前阵子在拆一个代码智能体的核心实现时，越看越觉得那套东西妙得厉害: 没有堆太多花活，真正起作用的部分反而非常收敛，工具发现、schema 转换、模型决策、脚本执行，一层扣一层，简洁得近乎锋利。看完之后很难不手痒，忍不住想自己做一个更小、更快、也更贴近日常终端使用习惯的版本。

于是有了 `oh-my-tai`。

它不打算变成一个无所不能的大而全 agent，也不打算接管你的开发流程。它想做的事情很单纯: 让你用一句自然语言，重新唤起你自己的脚本工具箱。

## 它是什么

`oh-my-tai` 是一个极简的 LLM CLI 工具。

你把常用脚本放进一个目录里，给它们加上少量注解；`tai` 在启动时扫描这些脚本，把它们转换成模型可理解的工具定义，然后让模型决定应该调用哪个脚本、带什么参数，最后直接执行。

一句话说，就是:

> 用自然语言，调你自己写过的脚本。

适合这样的场景:

- 你有很多零散但有用的小脚本
- 你经常忘记脚本名、参数名或调用方式
- 你想要的是“快点帮我执行掉”，不是一个很重的通用 agent 框架

## 为什么它很轻

这个项目最有意思的地方，不是功能多，而是刻意克制。

它几乎把结构压到了不能再薄的程度:

- `config.py`: 读取和生成 `~/.tai/config.toml`
- `tools.py`: 扫描工具目录，解析脚本头部注解
- `llm.py`: 用标准库发起 OpenAI 兼容接口请求，处理 tool calling
- `executor.py`: 执行脚本，收集输出，做最基本的危险命令拦截
- `cli.py`: 串起单次调用和交互模式

整个调用链也非常短:

1. 启动 CLI
2. 读取配置
3. 扫描工具目录中的可执行脚本
4. 把脚本注解转成函数调用 schema
5. 把用户输入交给模型
6. 执行模型选择的脚本
7. 把执行结果回填给模型，直到拿到最终回答

轻量主要来自这几件事:

- 纯 Python 实现，没有引入重型框架
- LLM 请求直接走 `urllib`，没有额外 SDK 依赖
- 工具定义不需要单独写 JSON，只靠脚本头部注解
- 默认就是本地脚本执行，不额外包一层复杂的 runtime
- agent loop 很短，目标明确，不做过度抽象

所以它的定位很清楚:

- 不是通用自主代理
- 不是工作流编排平台
- 不是要替代 shell
- 它就是一个“用自然语言调用脚本收藏”的超轻入口

## 适合拿它做什么

- 给自己那堆 `~/bin`、`scripts/`、`dotfiles` 里的工具加一个自然语言入口
- 在记不住脚本名时，快速把意图翻译成实际执行
- 做一个足够小、足够清楚的 agent harness 学习样本

如果你想要的是大规模规划、复杂多工具协作、长链路自治，那不是这个项目的目标。这个项目更像一把很小但很顺手的刀。

## 安装

项目使用 `uv` 管理。

```bash
uv sync
```

本地开发运行:

```bash
uv run tai "帮我做某件事"
```

也可以安装成命令:

```bash
uv pip install -e .
```

然后直接执行:

```bash
tai "列出当前目录下所有 Python 文件"
```

## 首次启动会发生什么

第一次运行 `tai` 时，程序会自动创建:

- `~/.tai/config.toml`
- `~/.tai/tools/`

默认配置大致如下:

```toml
default_provider = "openai"

[providers.openai]
api_key = ""
base_url = "https://api.openai.com/v1"
model = "gpt-4o-mini"

[tools]
directory = "/Users/your-name/.tai/tools"
```

你至少需要补上 `api_key`。如果你使用兼容 OpenAI Chat Completions 的服务，也可以改 `base_url` 和 `model`。

建议执行:

```bash
chmod 600 ~/.tai/config.toml
```

## 如何添加自己的工具

`oh-my-tai` 会递归扫描工具目录中的“可执行文件”，并读取文件顶部的注解。

一个最小示例:

```bash
#!/bin/bash
# @name weather
# @desc Query weather for a city
# @param city:string:City name

echo "Weather for $2"
```

保存到 `~/.tai/tools/weather.sh` 后，记得加执行权限:

```bash
chmod +x ~/.tai/tools/weather.sh
```

这里有几个约定:

- 只有带 `@name` 的脚本才会被注册成工具
- 只扫描文件开头连续的注释区；遇到第一行非注释内容就停止解析
- `@param` 格式是 `name:type:description`
- 当前参数会被转成命令行形式: `--参数名 参数值`
- 布尔参数在值为 `true` 时会变成单独的 `--参数名`

例如模型决定调用:

```text
weather(city="Shanghai")
```

实际执行时会变成:

```bash
~/.tai/tools/weather.sh --city Shanghai
```

所以写脚本时，要按这个参数传递方式处理输入。

## 怎么用

单次调用:

```bash
tai "帮我查一下上海天气"
```

交互模式:

```bash
tai -i
```

交互模式下会保留对话历史，你可以连续下指令，输入 `exit`、`quit` 或 `q` 退出。

## 当前执行模型

项目内部是一个非常短的 loop:

- 先把用户意图和工具列表发给模型
- 如果模型返回 tool call，就执行对应脚本
- 把脚本输出回传给模型
- 重复最多 5 轮，直到模型给出最终文本响应

脚本执行时会:

- 捕获 `stdout` 和 `stderr`
- 返回退出码
- 默认 120 秒超时
- 拦截少数明显危险的命令模式，比如 `rm -rf /`、`sudo`、`curl ... | bash`

这里的安全策略是“最基础的兜底”，不是完整沙箱。也就是说，`tai` 追求的是轻和快，不是强隔离执行环境。工具脚本本质上仍然代表你的本地权限边界。

## 项目结构

```text
src/tai/
  cli.py        # CLI 入口，单次执行与交互模式
  config.py     # 配置加载与默认配置生成
  tools.py      # 工具扫描与注解解析
  llm.py        # OpenAI 兼容接口调用与 tool calling loop
  executor.py   # 脚本执行、超时与基础安全检查
tests/
  ...           # 对各模块的单元测试
```

## 总结

`oh-my-tai` 不是为了把事情搞复杂，恰恰相反，它是因为看过一套很漂亮的极简实现之后，反而更想把问题重新收束到最小。

如果你也喜欢那种“小得刚刚好”的工具，这个项目应该会顺手。它不试图替你思考一切，它只是把一句自然语言，尽快接到你自己的脚本能力上。
